레지스트리는 독립 Package와 App을 발견하고 검토하는 색인이다. 플랫폼은 이 색인을 바탕으로 조합, 해석, 설치, 갱신을 연결한다. upstream Package의 내용을 가져와 새 소유권 아래 재배포하는 artifact host를 초기 전제로 삼지 않는다.

## Registry 범위

- personal registry: 개인이 선택한 source와 비공개 App 구성을 관리한다.
- public registry: 공개 upstream Package와 공개 App manifest를 찾을 수 있게 한다.
- team registry: 조직이 승인한 source, policy, 내부 Package와 App을 공유한다.

같은 Package가 여러 registry에 나타날 수 있다. identity는 registry 표시 이름이 아니라 canonical source와 source path, resolved revision을 기준으로 판별한다. registry entry는 upstream metadata의 snapshot과 검증 결과를 담을 수 있지만 원본을 가장하지 않는다.

## 탐색과 검토

웹과 CLI에서 다음 정보를 확인할 수 있어야 한다.

- 종류, 유지관리 source, license, 최근 해석 상태
- export된 component와 dependency graph
- host 및 runtime compatibility
- 요구 permission과 executable surface
- App이 선택한 Package와 exact lock 상태
- source preview와 closure에 포함되는 파일

검색은 이름뿐 아니라 종류, host, capability, license, permission으로 좁힐 수 있어야 한다. preview는 신뢰되지 않은 script나 MDX를 실행하지 않고 정적 내용으로 보여준다.

UI primitive와 block은 텍스트 목록만으로 비교하기 어렵다. visual catalog는 격리된 preview, 필요한 style과 asset, component dependency를 함께 보여준다. preview 환경의 network와 script 권한은 실제 설치 권한과 별도로 제한한다.

## Composer 경험

사용자는 bowl에 Package component를 추가하고 다음 순서로 App을 구체화한다.

1. 목적과 target host를 정한다.
2. 독립 Package와 component를 선택한다.
3. resolver가 계산한 dependency와 충돌을 본다.
4. source, license, permission, capability gap을 검토한다.
5. manifest와 lock 후보를 저장한다.
6. 로컬 host에 설치하거나 내보낸다.

Composer는 큰 framework를 하나의 카드로 보여줄 수 있지만 내부 Package 경계와 권한을 접지 않는다. App template도 독립 Package를 참조하는 manifest 예시로 취급한다.

## Resolver, cache, adapter

Resolver는 requested ref를 immutable commit으로 해석하고 component closure와 transitive dependency를 계산한다. policy를 통과하지 못하거나 외부 참조를 안전하게 해석할 수 없으면 fail closed한다.

Content-addressed cache는 같은 closure의 재사용, offline materialization, hash 검증을 돕는다. cache hit 여부가 resolution 의미를 바꾸어서는 안 된다. upstream이 사라진 경우 cache가 임시 복구 수단이 될 수 있지만 라이선스와 재배포 권한을 넘어서지는 않는다.

Adapter는 해석된 App을 host-native output으로 변환한다. 초기에는 한 host에 집중하고 이후 Cursor, Claude Code, Codex로 넓힐 수 있다. 각 adapter는 지원 capability와 손실되는 기능을 machine-readable report와 사람이 읽는 설명으로 함께 제공하는 방향을 검토한다.

## 신뢰와 moderation

레지스트리 등재는 안전 보증과 같지 않다. 검증 상태, maintainer claim, source ownership 확인, 서명 여부, license 확인, 신고와 차단 이력을 구분해 보여준다. public registry에는 악성 package 신고, source 변경 감지, 이름 사칭 완화, 검색 노출 제한이 필요하다. team registry는 허용 source, license, permission, publisher policy를 별도로 적용할 수 있다.
