이 문서의 이름은 설계 초안을 위한 것이다. 실제 schema field나 CLI 명령으로 확정된 것은 아니다.

## 용어

### Package

독립적으로 versioning되고 source를 추적할 수 있는 배포 단위다. 하나 이상의 component를 export한다. 개념적 크기와 packaging format은 별개다. 작은 skill도, 여러 도구를 묶은 framework pack도 Package일 수 있다.

### Component

Package가 외부에 제공하는 선택 가능한 기능 단위다. skill, agent, instruction, UI block, CLI, MCP server, scaffold recipe 등이 될 수 있다. component를 설치할 때는 본문 파일뿐 아니라 참조하는 script, reference, template, asset까지 closure에 포함한다.

### Plugin

특정 host에서 기능을 활성화하고 배포하기 위한 envelope다. Claude Code plugin처럼 host 규칙에 맞춰 여러 component를 담을 수 있다. Plugin은 개념적 크기를 뜻하지 않으며 Package나 App의 동의어가 아니다.

### App

사용 목적에 맞춰 여러 독립 Package와 component를 조합한 최상위 구성이다. App은 manifest와 lock으로 재현되며, 구성 Package의 upstream 소유권이나 갱신 주기를 넘겨받지 않는다.

### Bowl 또는 Composition

App을 만들기 위해 Package와 component를 선택하고 정책을 적용하는 작업 공간을 가리키는 잠정 이름이다. bowl은 새 source identity를 만들지 않는다. 저장된 구성이 배포 가능한 단위가 되면 App으로 다룬다.

### Registry

Package와 App의 metadata, source, version, compatibility, dependency를 색인하는 서비스다. 기본 역할은 upstream 내용을 복제해 소유하는 것이 아니라 발견과 검토를 돕는 것이다.

### Scaffold

template과 recipe로 사용자 소유 파일을 생성하거나 기존 프로젝트에 구조를 추가하는 lifecycle operation이다. Package 종류나 Plugin 형식이 아니다. 생성 뒤 수정된 파일은 단순 교체 대상이 아니다.

### Adapter

해석된 App을 Cursor, Claude Code, Codex 같은 host의 네이티브 구조로 materialize하는 변환 계층이다. host가 요구 기능을 지원하지 않으면 누락을 숨기지 않고 capability gap을 보고한다.

### Manifest

사용자가 원하는 Package, component, version/ref 범위, target host, permission 및 license policy를 적는 선언이다. desired state를 표현하며 정확한 commit이나 설치 경로를 모두 담는 기록은 아니다.

### Lock

resolver가 선택한 canonical URL, requested ref, resolved commit, source path, dependency graph, closure hash 등 exact resolution을 기록한다. 같은 manifest가 시간이 지나며 다른 결과를 내지 않도록 기준점을 제공한다.

### Provenance

생성되거나 materialize된 artifact와 함께 이동하는 출처 기록이다. 원본 component, resolved commit, 입력 hash, adapter와 생성 방식의 관계를 추적한다.

### Receipt

특정 machine에 실제로 설치된 경로, materialized hash, host 상태, 보존한 config나 secret, 설치 시각을 기록하는 local state다. 다른 machine과 공유하는 lock을 대신하지 않는다.

## 계층과 포장 형식

생태계의 기능 계층은 다음처럼 구분할 수 있다.

| 계층 | 주된 역할 | 예시 |
|---|---|---|
| utility skill | 여러 도메인에서 반복되는 작은 능력 | UI 개선, 문서 검토 |
| domain/workflow skill | 특정 분야의 판단과 절차 | 기업 분석, 릴리스 점검 |
| plugin/framework | 여러 component의 host 활성화 또는 개발 규약 | host plugin, framework pack |
| scaffolder/starter | 프로젝트나 기능의 초기 구조 생성 | dashboard starter |
| platform/studio | 조합, 실행, 관찰, 편집 | registry, visual studio |

이 표는 source 경계나 설치 단위를 결정하지 않는다. 하나의 framework가 여러 Package를 참조할 수도 있고, 하나의 Package가 여러 계층의 component를 export할 수도 있다.

## 식별 원칙

표시 이름은 전역 identity가 아니다. 같은 이름이 서로 다른 owner와 source에 존재할 수 있다. 식별에는 canonical source URL, source 내부 경로, 요청한 ref와 해석된 commit이 필요하다. 구체적인 기록 항목은 [패키지 모델](package-model)과 [보안과 출처 증명](security-and-provenance)에서 다룬다.
