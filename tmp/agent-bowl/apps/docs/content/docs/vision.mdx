## 생태계에 대한 관점

에이전트 생태계에는 이미 서로 다른 크기와 역할의 자산이 있다. 작은 수평형 utility skill, 특정 도메인이나 업무 흐름을 담은 skill, host plugin과 framework, starter와 scaffolder, dashboard와 studio가 각자의 저장소와 배포 방식으로 발전한다. 문제는 이들을 하나의 형식으로 다시 포장하는 데 있지 않다. 출처와 경계를 보존한 채 발견하고 함께 쓸 수 있는지 판단하고 재현 가능한 App으로 조합하는 기반이 필요하다.

계층은 대략 다음과 같이 볼 수 있다.

1. `improve-ui` 같은 재사용 utility skill
2. 회계, 조사, 검토처럼 도메인이나 workflow를 담은 skill
3. 여러 지침과 실행 도구를 묶는 plugin 또는 framework
4. 프로젝트 생성을 맡는 scaffolder와 starter
5. 실행, 관찰, 편집을 함께 제공하는 platform과 studio

이 순서는 가치나 포장 형식의 서열이 아니다. 작은 패키지도 독립적으로 배포될 수 있고, 큰 framework도 여러 패키지의 App 조합으로 표현될 수 있다.

Toolcraft와 비슷한 framework pack을 예로 들면 agent guidance, component library, scaffold recipe, CLI나 MCP, verification asset, dashboard 또는 studio를 함께 제공할 수 있다. HyperFrames 같은 수직형 agent-native framework도 skill, CLI/engine, catalog, studio, rendering을 결합할 수 있다. `agent-bowl`은 이런 프로젝트의 내용을 흡수하지 않고 각 요소의 출처와 결합 관계를 드러내는 쪽을 택한다.

## 컴파일러 비유

에이전트는 제한적으로 “확률적 compiler driver와 build system/runtime의 결합”으로 볼 수 있다.

- 사용자 prompt는 높은 수준의 의도다.
- instruction과 skill은 판단을 돕는 cognitive dependency다.
- CLI와 MCP server는 실행 가능한 dependency다.
- template과 component는 결과물에 들어가는 material dependency다.
- dashboard와 service는 운영 단계의 dependency다.

이 비유는 입력, 의존성 해석, materialization, 실행, 검증을 나누는 데 유용하다. 그러나 에이전트는 결정론적 compiler가 아니다. 같은 입력에서도 결과가 달라질 수 있고 자연어 지침은 정적 type이나 ABI처럼 완전하게 검사되지 않으며 외부 도구와 모델의 상태도 결과에 영향을 준다. 따라서 lock과 hash는 입력 재료를 고정할 수 있을 뿐 실행 결과의 완전한 동일성을 보장하지 않는다.

## 지금 필요한 이유

패키지 종류와 host가 늘수록 설치 안내를 복사하는 방식은 출처, 권한, 버전, 호환성을 놓치기 쉽다. 이름만 같은 패키지를 구별하기 어렵고, scaffold가 생성한 코드와 package manager가 관리하는 파일의 갱신 규칙도 서로 다르다. 로컬 구성을 cloud로 옮길 때 이 차이를 다시 정의하면 같은 App이라는 보장도 약해진다.

필요한 것은 다음을 한 흐름에서 다루는 얇은 조정 계층이다.

- 원본 저장소를 기준으로 한 탐색과 식별
- 패키지 및 App 의존성 해석
- 정확한 source ref와 component closure 고정
- 권한과 라이선스 검토
- host별 네이티브 설치
- 소유권에 맞는 갱신과 제거
- 로컬과 cloud가 공유하는 구성 계약

## 원하는 결과

사용자는 레지스트리에서 독립 패키지를 비교하고 필요한 패키지를 bowl에 담아 App을 구성하며 설치 전에 전체 의존성·출처·권한을 확인할 수 있어야 한다. 설치 뒤에는 어떤 원본의 어느 commit이 어느 경로에 놓였는지 추적할 수 있어야 한다. 유지관리자는 자신의 저장소와 릴리스 주기를 바꾸지 않고 색인될 수 있어야 한다.

이 목표를 위한 용어와 경계는 [핵심 개념](concepts), 데이터 책임은 [패키지 모델](package-model)과 [App 모델](app-model)에서 이어서 설명한다.
