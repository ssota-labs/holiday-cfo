App은 하나의 사용자 목적을 위해 독립 Package의 component를 조합한 최상위 단위다. 구성 Package는 각자의 canonical source와 release lifecycle을 유지한다. App 저장소가 Package 내용을 복사해 소유하는 구조를 전제로 하지 않는다.

## 구성 기록의 역할

| 기록 | 답하는 질문 | 공유 범위 |
|---|---|---|
| manifest | 무엇을 원하고 어떤 policy를 적용하는가 | App과 함께 공유 |
| lock | 어떤 source와 commit, closure로 해석했는가 | App과 함께 공유 |
| provenance | 생성물은 어떤 입력과 변환에서 왔는가 | artifact와 함께 이동 |
| receipt | 이 machine에 무엇이 어디에 설치되었는가 | machine-local |

manifest에는 Package와 component 선택, 허용 version/ref, target host, permission 및 license policy를 둔다. lock에는 exact source resolution과 dependency graph를 둔다. provenance는 scaffold 산출물과 adapter 결과에도 남는다. receipt는 실제 경로와 materialized hash, 보존한 local config 상태를 기록한다.

content-addressed cache는 다운로드와 검증을 줄이고 사라진 작업 tree를 다시 materialize하는 데 도움을 준다. 그러나 cache는 source identity나 lock을 대신하지 않는다. symlink는 구현 최적화로도 신중히 다뤄야 하며 설치 계약에는 포함하지 않는다.

## 조합과 해석

사용자는 registry에서 Package를 찾아 bowl에 component를 담고 App manifest 후보를 만든다. resolver는 dependency closure, source identity, compatibility, permission, license policy를 확인한 뒤 lock 후보를 제시한다. 사용자가 검토하고 승인하면 adapter가 target host의 네이티브 파일과 설정을 만든다.

Cursor, Claude Code, Codex는 같은 기능을 서로 다른 구조로 활성화할 수 있다. adapter는 공통의 가상 형식을 억지로 주입하기보다 host-native output을 만든다. hook, UI, MCP lifecycle처럼 표현할 수 없는 기능은 빠뜨린 채 성공으로 처리하지 않고 gap을 보고한다.

## Lifecycle

이름은 설명을 위한 잠정 구분이며 CLI 명령으로 확정되지 않았다.

### Install

lock을 검증하고 closure를 가져온 뒤 권한을 확인한다. adapter가 materialize한 결과의 hash와 설치 경로를 receipt에 남긴다. 기존 사용자 파일과 충돌하면 먼저 소유권을 판정한다.

### Create 또는 Scaffold

recipe와 template로 새 사용자 소유 코드를 만든다. 생성 시점의 base provenance를 남겨 이후 migration과 three-way merge에 쓴다. 생성 뒤 편집된 코드는 pack-managed replace 대상이 아니다.

### Update

manifest 범위 안에서 새 resolution을 계산하고 이전 lock과 source, dependency, license, permission 차이를 보여준다. executable capability가 늘면 새 승인을 받는다. 승인 후 artifact ownership mode에 맞춰 교체, package-manager update, deploy, migration 중 하나를 수행한다.

### Migrate

generated user-owned code에는 version 간 migration이나 three-way merge를 적용한다. 자동 병합이 안전하지 않으면 충돌과 필요한 작업을 표시하고 원본을 보존한다.

### Verify

source hash, closure, materialized output, host capability, receipt drift를 확인한다. Package가 제공한 verification asset은 출처와 실행 권한을 검토한 뒤 사용한다.

### Uninstall

receipt를 기준으로 pack-managed artifact만 제거한다. 사용자 소유 코드, secret, 공유 dependency는 자동 삭제하지 않고 남은 항목을 보고한다.

## Local과 cloud

로컬 설치와 향후 cloud 실행은 같은 App manifest와 lock을 사용한다. cloud는 hosted storage, scheduler, secret manager, queue와 worker, notification, execution 환경을 더할 수 있다. 이 운영 기능은 Package의 의미나 source identity를 다시 정의하지 않는다. 동일한 lock을 사용해도 모델, 외부 API, 시간 의존 입력 때문에 실행 결과가 항상 같아지는 것은 아니다.
