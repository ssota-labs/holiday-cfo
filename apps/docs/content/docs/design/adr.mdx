type: decision
summary: 거부한 대안이 본문이다. 코드를 읽어선 알 수 없는 부분이라서.
---

**거부한 대안이 각 항목의 본문이다.** 무엇을 골랐는지는 누구나 코드를 읽어서 알 수
있다. 무엇을 시도했다 버렸는지는 아무도 못 읽고, 그게 정확히 1년 뒤에 다시 제안되는
것이다. "환율을 그냥 저장하면 안 되나?"에는 답이 있고, 그 답은 누군가 원장을 다시
쓰기 전에 찾을 수 있는 곳에 있어야 한다.

`status`는 되돌릴 수 있는지를 정직하게 말한다 — `accepted`는 바뀔 수 있고, `locked`는
과거를 다시 쓰는 마이그레이션 없이는 못 바꾼다.

>
DDL은 Drizzle이 만들고 읽기는 `Db` 래퍼가 한다. `integer()`가 뱉는 DDL은 평범한
`INTEGER`라 이 분리는 비용이 0이다.

**도입 동기였던 전제는 성립하지 않았다.** Drizzle은 SQLite와 Postgres에 하나의 스키마를
주지 않는다 — `sqliteTable`과 `pgTable`은 별개 빌더고 컬럼 타입도 안 겹친다. Supabase는
어차피 자기 스키마를 쓴다. SQLite의 `RAISE(ABORT)`와 Postgres의 plpgsql `RAISE
EXCEPTION`은 방언이 다른 거지 툴체인 문제가 아니고, 포트 경계가 이미 그걸 허용한다.

**진짜 얻은 건 해시 추적이다.** `__holiday_migrations`가 각 마이그레이션의 sha256을
기록해서, 적용된 걸 수정하면 다음 열기에서 터진다. Drizzle의 `__drizzle_migrations`에서
가져온 아이디어인데 테이블은 우리 것이다 — Drizzle의 `migrate()`는 런타임에 디스크에서
`.sql`을 읽고, 이 CLI는 옆에 아무것도 없는 단일파일이다.

**대가:** STRICT 테이블을 잃었다(drizzle-kit 미지원). 손편집으로 INTEGER 컬럼에 TEXT를
넣는 걸 막던 ring 3 하나가 빠졌다. 그리고 RC(`1.0.0-rc.4`)에 핀됐다 — stable에는
node:sqlite 드라이버가 없다.
</Decision>

<Decision
  id="ADR-006"
  title="스토어는 하나. 방언만 갈라진다"
  status="accepted"
  date="2026-07-17"
  commit="2c90bef"
  rejected={[
    { option: 'store-sqlite를 기계 변환해서 store-postgres 만들기 (착수 직전까지 갔다)', because: '쓰는 날에는 두 파일이 일치하지만, 한쪽에만 들어간 첫 손수정에서 갈라진다. 포트 경계를 둔 이유가 정확히 그걸 막으려는 건데 그 경계를 배신하는 방법이었다.' },
    { option: 'Drizzle로 스키마 하나 두고 두 엔진에 공유', because: 'ADR-005. sqliteTable과 pgTable은 별개 빌더고 트리거 본문은 진짜로 다른 SQL이다. 스키마는 공유가 불가능하다 — 하지만 그게 스토어까지 갈라야 한다는 뜻은 아니었다.' },
    { option: 'node:sqlite에 맞춰 스토어를 동기로 유지', because: 'Postgres를 동기로 만들 방법이 없다. 동기 엔진에 비동기 외투를 입히는 건 값이 싸고(모든 호출이 즉시 resolve된다) 반대 방향은 불가능하다.' },
  ]}
>
전제를 믿지 않고 **셌다.** 스토어 1,486줄 중 SQLite 고유인 것:

| | |
|---|---|
| `GLOB` | 3 |
| `PRAGMA` | 3 |
| `INSERT OR IGNORE` | 1 |
| `BEGIN IMMEDIATE` | 1 |
| **합계** | **8** |

여덟 줄. 구조적 차이는 `node:sqlite`가 동기라는 것 하나뿐이었고, 그건 **드라이버
인터페이스**를 두라는 근거지 스토어를 하나 더 쓰라는 근거가 아니다.

그래서 `@holiday-cfo/store-sql`이 `LedgerStore`를 하나만 구현하고, 엔진 패키지는 드라이버·
스키마·마이그레이션만 가진다. 8곳은 `Dialect`로 접힌다 — `subtreeWildcard()`와
`prepare()`, 그리고 엔진의 선택적 `init`/`checkpoint` 훅.

**`GLOB`이 공유 SQL에 남아있는 건 SQLite-ism을 흘린 게 아니다.** 이 매칭은 대소문자를
구분해야 하는데 SQLite의 `LIKE`는 구분하지 않는다 — `Assets:bank:x`가 `Assets:Bank`
접두사에 걸린다. `GLOB`이 SQLite에서 그걸 쓰는 철자이고, Postgres에서는 `LIKE`가 이미
대소문자를 구분한다. 같은 의미, 두 방언. 드라이버가 번역한다.

**이 리팩터를 판단 가능하게 만든 건 적합성 스위트다.** 1,486줄을 옮기고 전부 비동기로
바꿨는데 18개가 그대로 통과했다. 스위트가 없었으면 "아마 같을 것"이 최선이었다.

부산물로 타입체커가 잡아준 것: `await this.db.all<T>(...).map(...)`은
`await (….map(…))`으로 파싱돼서 `.map`이 Promise에 붙는다. 전 호출부에 괄호가 필요했고
20군데가 한 번에 드러났다.
</Decision>
