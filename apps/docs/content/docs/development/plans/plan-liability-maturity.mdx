## 목표

[prd-liability-maturity](/docs/planning/prds/prd-liability-maturity)의
**유동·비유동 부채 합계**를 기준일에 계산해 `balance`와 `close`(dry-run 포함)에
붙인다. 계정 트리·원장 스키마는 그대로다.

계약: [SPEC-liability-maturity](/docs/domain/liability-maturity).

## 영향 범위

| 영역 | 변경 |
|---|---|
| `packages/core` | 만기 구분 순수 함수 + 도메인 테스트 (카드/할부/대출/잔액불일치) |
| `packages/cli` | `balance`·`close` 출력·JSON에 요약 추가, `note()` 한 줄 |
| `apps/docs` | CLI 스펙(`balance`/`close`)·용어집·기능 페이지 정합 |
| 플러그인 | “유동부채” 질의 시 위 명령을 읽도록 스킬/reference 한 줄 |

`store-*` 마이그레이션·테이블 없음.

## 구현 순서

1. core: `classifyLiabilityMaturity(asOf, balances, schedules)` —
   창 `(asOf, asOf+12m]`, 원금만, 잔액 불일치 규칙(SPEC)
2. 도메인 테스트로 SPEC 수락 기준 1–3 고정
3. usecase: 잔액·스케줄 로드 후 core 호출
4. CLI `balance`: 부채가 결과에 포함될 때 요약(+ 계정별) 출력
5. CLI `close` / `--dry-run`: 기간 말일 기준 같은 요약
6. docs CLI 스펙·glossary 용어(유동부채/비유동부채)
7. 플러그인: 질의 → `balance --as-of` / 마감 결과 읽기
8. `pnpm --filter @holiday-cfo/core test` · 관련 CLI 테스트 · `check:skill`

## 검증 계획

### 자동 테스트

- core: 카드 전액 유동
- core: 18개월 할부 → 12+6
- core: 스케줄 < 잔액 → 부족분 유동
- core: 스케줄 > 잔액 → 비율 축소 후 합 = 잔액
- close dry-run JSON과 balance `--as-of` 말일 합계 동일

### 수동 테스트 / 기획 체크

1. 대출+카드 scratch 장부에서 잔액 조회 → 유동·비유동·총액
2. 같은 날 마감 dry-run → 숫자 일치
3. 계정 목록에 유동/비유동 루트 없음
4. “유동부채 얼마야?” 스킬 경로로 같은 숫자

## 마이그레이션과 호환성

없음. JSON에 필드 **추가**만. 기존 키 의미 유지.

## 위험과 롤백

| 위험 | 대응 |
|---|---|
| 스케줄이 낡은데 잔액은 맞음 | 잔액 우선 + 부족분 유동 (보수) |
| 거치·만기일시 대출 | 창 밖 원금은 비유동 — bullet 만기일이 창 밖이면 전액 비유동 |
| 출력 소음 | 부채 잔액이 0이면 요약 생략 가능(스펙과 CLI에서 동일 규칙) |

롤백: CLI 출력·core 함수 제거. 데이터 이주 없음.
