## 목표

현금흐름의 카드 이용분 줄이 **그 주기 실제 이용·환불**만 담게 한다.
카드대금 결제를 draft로 승인하다 카드가 이중으로 잡히고, 그 이중분을 되돌리는
정정까지 이용분으로 잡혀 청구액이 수백만 원 단위로 부푸는 일을 없앤다.

계약: [SPEC-cashflow-card-bill](/docs/domain/cashflow-card-bill).

## 영향 범위

| 영역 | 변경 |
|---|---|
| `packages/core` `domain/cashflow.ts` | `projectCardBills` 제외 규칙 확장 |
| `packages/core` `usecases/cashflow.ts` | 투영 입력에 정정·시스템 종류 전달 |
| `packages/store-sql` (필요 시) | posting 스트림 또는 조인으로 txn 메타 노출 |
| `packages/core` `cashflow.test.ts` | 정정·개시·자본 상대 제외, 이용·환불·결제는 유지 |
| `apps/docs` `policy.mdx` | 구현 PR에서 POLICY-022 + 테스트 링크 (이 기획 PR에서는 Rule 추가 안 함 — `check:rules`가 테스트 존재를 요구) |
| `apps/docs` 기능 문서 | 수락 기준을 스펙에 맞춤 (이 PR) |

## 구현 순서

1. `ProjectionPosting`(또는 동등 입력)에 `correctsTxnId` · `systemKind`를 넣는다.
   posting 스트림에 없으면 txn 조인·별도 조회로 채운다.
2. `projectCardBills`에서 기존 funding 제외에 더해 다음을 뺀다.
   - `correctsTxnId != null`
   - `systemKind != null`
   - 카드 밖 다리가 Equity뿐인 전표 (계정 타입/코드 prefix로 판별)
3. 도메인 테스트: 이중반영 취소 정정(카드 − / 자본 +)이 이용분에 안 잡힘;
   가맹점 이용·환불·정상 결제는 기존과 같음; 개시(`opening_balance`) 제외.
4. `POLICY-022`를 policy.mdx에 추가하고 위 테스트에 링크한다.
5. `pnpm --filter @holiday-cfo/core test` · docs `check:rules` · `check:planning`.

## 검증 계획

### 자동 테스트

- 주기 안 `Card −` / `Equity +` 정정만 있으면 이용분 줄이 나오지 않는다
- `correctsTxnId`가 있는 정정은 상대와 무관하게 이용분에서 빠진다
- `systemKind: opening_balance` 카드 다리는 이용분에서 빠진다
- 가맹점 이용은 결제일에 그대로 잡힌다
- funding 결제는 계속 이용분에서 빠진다 (POLICY-004)
- 환불은 이용분을 줄인다
- 할부 스케줄 줄은 이용분과 별개로 같은 결제일에 남는다

### 수동 테스트 / 기획 체크

1. 재현: 청구주기 첫날에 카드 − / 자본 + 수기 전표만 두고 `cashflow` → 그 주기
   이용분 줄이 없어야 한다
2. 같은 주기에 가맹점 이용을 넣으면 이용분은 그 금액만이다
3. 할부가 있으면 결제일에 할부 줄만 따로 보인다

## 마이그레이션과 호환성

- DB 마이그레이션 없음
- 과거 전표는 수정하지 않는다. 현금흐름 숫자만 달라질 수 있다
- 정정 링크 없이 제목만 `정정:`인 수기 전표는 **자본 상대** 규칙으로 걸러진다.
  상대가 비용·자산이면 이용분에 그대로 남을 수 있다 — 그때는 정정 링크를
  쓰도록 에이전트 안내를 후속으로 보강한다 (이 계획 범위 밖)

## 위험과 롤백

| 위험 | 대응 |
|---|---|
| Equity 상대를 쓰는 드문 정상 이용이 제외됨 | 개인 장부에서 카드↔자본 이용은 사실상 개시·정정. 사례 있으면 스펙을 좁힌다 |
| posting 스트림 확장 누락으로 메타가 비어 제외가 안 됨 | 통합 경로 테스트로 usecase까지 고정 |
| POLICY Rule을 테스트 없이 추가해 docs build 실패 | Rule은 구현 PR에서만 추가 |

롤백: `projectCardBills`와 투영 입력 매핑을 되돌리면 된다. 원장 데이터 손상 없음.
