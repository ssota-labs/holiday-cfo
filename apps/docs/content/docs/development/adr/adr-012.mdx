문서 사이트에서 가져올 것은 **엔진 전체**가 아니라 **MDX 작성 계약**이다.

채택하는 것 (docs와 같은 디자인·타입 층):

1. 페이지는 MDX 파일 트리로 둔다
2. 커스텀 칸은 페이지마다 import하지 않고 **전역 레지스트리**에 등록한다
   (`getMDXComponents`와 같은 모양)
3. 타입은 `MDXComponents`(`mdx/types`) 위에 닫힌 어휘를 얹는다
4. frontmatter는 Zod 스키마로 계약한다 (dash용 필드. Diátaxis 문서 타입을
   그대로 복사할 필요는 없다)
5. 그 레지스트리에 **대시보드 칸**(`CashRunway`, `BalanceTable`, …)을 추가한다

이식하지 않는 것:

- `fumadocs-ui` DocsLayout·사이드바·문서 크롬
- `apps/docs` Next + `fumadocs-mdx` postinstall 컬렉션을 사용자 `dash/`의
  런타임으로 복사
- 문서 전용 블록(`Rule`, `SchemaTable`, …)을 장부 화면 어휘에 섞기

런타임은 vinext 템플릿과 `@holiday-cfo/blocks`를 유지하고, 숫자는
`holiday dash data` 스냅샷만 읽는다. MDX 컴파일러를 Vite 플러그인으로 둘지,
얇은 `fumadocs-mdx` 호환 층을 둘지는 구현 스파이크에서 고른다. 어느 쪽이든
**관찰 가능한 계약은 위 등록·타입 모델**이다.

[ADR-008](/docs/development/adr/adr-008)의 로컬 조종석 / 배포 사진 경계는
그대로다. 쓰기는 카탈로그에 올리지 않고, 로컬 API → CLI만 통과한다.
</Decision>
