    { option: 'drizzle-orm으로 쿼리까지 전부 쓰기', because: 'Drizzle SQLite 세션은 bigint를 켜지 않고, integer를 number(53비트)로 본다. 2^53을 넘는 금액은 바로 터지고 고칠 훅이 없다. 이 장부 금액은 i64이고 테스트가 2^53+1 왕복을 못 박는다.' },
    { option: "bigint를 BLOB 문자열로 저장", because: '정확히는 남지만 SUM()이 안 되고 정렬도 바이트 순이라 잔액 조회가 깨진다.' },
    { option: '스키마 SQL을 손으로만 관리', because: '마이그레이션 추적이 약하고, “이미 적용한 파일을 고치지 말 것”이 주석에만 있었다.' },

>
테이블 정의·마이그레이션 파일은 **drizzle-kit**이 만든다. 실제 SELECT/INSERT는
우리 `Db` 래퍼가 한다. Drizzle이 만드는 DDL은 평범한 `INTEGER`라, 이렇게 나눠도
추가 비용은 거의 없다.

처음엔 “한 스키마로 SQLite와 Postgres를 같이”가 동기였다. 그건 **성립하지 않았다**.
`sqliteTable`과 `pgTable`은 별개고, 트리거 SQL도 방언이 다르다. 포트 경계가 이미
그걸 허용한다.

**실제로 얻은 것**은 마이그레이션 **내용 해시**다. `__holiday_migrations`에 sha256을
남겨, 이미 적용한 파일을 고치면 다음 열기에서 실패한다. Drizzle의 마이그레이션
테이블에서 아이디어를 빌렸지만, CLI는 디스크의 `.sql`을 런타임에 읽지 않는
단일 배포물이라 우리 테이블을 쓴다.

**대가.** STRICT 테이블은 못 쓴다(drizzle-kit 미지원). INTEGER에 TEXT를 넣는
실수를 DB가 덜 막아 준다. 또한 `node:sqlite` 드라이버가 있는 RC 버전에 핀한다.
</Decision>
