    { option: '플러그인에 단일파일 번들을 커밋 (원래 방식)', because: '`node:sqlite`를 강제하는 근거였는데(네이티브 애드온은 번들 불가), 대시보드를 붙이면서 사용자 프로젝트는 어차피 npm install을 한다는 게 분명해졌다. 번들을 유지할 이유가 CLI 자체 말고는 없어졌고, 커밋된 바이너리는 썩는다.' },
    { option: 'claude-code는 번들, codex는 npx (혼합)', because: '두 호스트가 서로 다른 CLI를 얻으면 한쪽에서만 나는 버그가 생긴다. 번들은 실행해야만 검증되는데(스코프 리네임이 번들을 조용히 깨뜨린 전례), 검증 표면을 둘로 늘릴 이유가 없다.' },
  ]}
>
CLI는 `@holiday-cfo/cli`로 npm에 배포한다. 두 플러그인은 스킬만 담고, 에이전트는
`npx @holiday-cfo/cli@latest`로 CLI를 얻는다. 첫 실행에 네트워크가 필요한 대신,
커밋된 바이너리도 재번들 CI도 없다.

**`node:sqlite` 선택은 그대로다.** 근거만 바뀌었다 — "네이티브 애드온은 번들 불가"에서
"의존성 0·네이티브 빌드 0이라 `npx` 첫 실행이 가볍다"로. 결론이 살아남았다는 건 그게
표면적 제약이 아니라 실제로 옳은 선택이었다는 뜻이다.

버전 정합성은 npm이 공짜로 준다: `holiday dash init`이 스캐폴드하는 프로젝트는
`@holiday-cfo/blocks`를 CLI 자기 버전으로 **정확히 핀**한다. CLI와 그것이 가르치는
블록 어휘가 한 버전으로 함께 움직인다.
</Decision>
