## 의존 방향

```
                  @holiday-cfo/core
         (도메인 + 포트 + 유스케이스)
                        ▲
        ┌───────────────┼───────────────────┐
        │               │                   │
  @holiday-cfo/     @holiday-cfo/     @holiday-cfo/
  contracts         adapters          store-sql
  (Zod JSON)        (HTTP·MCP I/O)   (LedgerStore 하나)
        ▲               ▲                   ▲
        └───────┬───────┘                   │
                │              ┌────────────┴────────────┐
                │              │                         │
           @holiday-cfo/cli  store-sqlite          store-postgres
           + deploy templates (로컬 SoR)            (Supabase/PGlite)
             (vercel-supabase / chatgpt-sites —
              벤더 SDK는 템플릿 package.json에만)
```

`core`는 어댑터에서 **아무것도** import하지 않는다. CLI와 생성된 BYOC 템플릿이
composition root다. Supabase / Vercel / vinext / D1 / R2 SDK는 **holiday 런타임
패키지에 넣지 않는다** — `packages/cli/templates/*`의 생성물 `package.json`에만 존재한다.

`@holiday-cfo/store-d1`은 projection으로만 존재하며 engine 적합성에 실패한다. ChatGPT
Sites는 그 결과가 바뀔 때까지 fax inbox + review/export로 제한한다 (능력 플래그로
실패를 숨기지 않음).

**`LedgerStore` 구현은 엔진마다 있지 않고 `store-sql`에 하나만 있다.** 엔진 패키지가
가진 건 드라이버, 스키마, 마이그레이션뿐이다. 이유는 취향이 아니라 측정이다 — 스토어
1,486줄 중 방언에 의존하는 건 8줄이었다. [ADR-006](/docs/dev/adr) 참조.

**별도 `@holiday-cfo/ports` 패키지는 없다.** 헥사고날에서 포트 인터페이스는 도메인의
소유다 — 바깥과 말하기 위한 도메인의 어휘. 따로 빼면 가짜 경계에 워크스페이스 엣지와
버전 동기화 비용만 붙는다. `package.json` `exports` 서브패스로 같은 강제력을 얻는다.

## 포트는 불균형을 표현할 수 없다

가장 중요한 성질이다. `insertPosting()` 같은 건 없다. 원자는 **검증된 거래 전체**다.

포트의 **유일한** 불변식 의무는 원자성과 내구성이다. 어댑터마다 균형을 재검증하면
어댑터 수만큼 미묘하게 다른 "균형"의 정의가 생긴다.

## 어댑터는 티어를 참칭할 수 없다

`init()`은 `tier === 'engine'`인데 세 보장이 전부 true가 아니면 **throw한다.**

**이게 "노션은 원장이 될 수 없다"를 README의 주장이 아니라 구조로 만든다.** 노션에서
3-leg 거래는 API 3콜이고, 2번과 3번 사이에서 실패하면 장부가 안 맞은 채로 남는다.
자세한 논거는 [ADR](/docs/dev/adr)에.

`streamPostings()`(필수 프리미티브) + `getBalances()`(capability 게이트 빠른 경로) 쌍이,
약한 백엔드를 **틀리게** 만드는 대신 **느리게** 만든다.

## 불변식 4중 링

| 링 | 위치 | 역할 |
|---|---|---|
| **1. 도메인 (권위)** | `Txn.create()` | 순수·동기·I/O 없음. 불균형 `ValidatedTxn`을 **반환할 수 없다**. mock 없이 property test 가능 |
| **2. 포트 (의무)** | `appendTxn()` | 유일한 의무는 원자성 + 내구성. 비즈니스 규칙 아님 |
| **3. 저장소 (심층방어)** | 트리거 (SQLite `RAISE(ABORT)` / Postgres plpgsql) | 링1의 버그, 캐스트, `sqlite3` 수동 편집을 잡음. engine 티어만 가능. 불변식 11개, 두 엔진 모두 |
| **4. 감사** | `holiday verify` | 전체 스캔 + 해시 체인 |

포트가 의도적으로 **가장 약한** 링이다. 포트는 기계적 보장(원자성)을 명세해야지
비즈니스 규칙(균형)을 명세하면 안 된다.

## 적합성 스위트가 경계를 진짜로 만든다

TypeScript 인터페이스는 문서고, **`runLedgerStoreConformance()`가 계약이다.**
18개 케이스가 **엔진마다** 돌고, 그중 셋이 위조하기 쉬운 것들을 증명한다:

- unit of work 중간 예외 → **0행** 기록
- 캐스트로 밀어넣은 불균형 거래 → 저장소가 거부
- `2^53+1` 왕복이 `number`로 뭉개지지 않음

**첫 어댑터보다 먼저 썼다.** 적합성 스위트는 굽힐 압력이 생기기 *전에* 존재해야만
경계를 정직하게 유지한다.

그리고 그게 값을 했다. `store-sql` 추출은 1,486줄을 패키지 밖으로 옮기고 전부 동기에서
비동기로 바꾼 리팩터인데, 18개가 **그대로** 통과했다. 이 스위트가 없었으면 "아마
같을 것"이라고 말할 수밖에 없었다. Postgres 어댑터도 같은 18개로 증명한다 — pglite는
목이 아니라 WASM으로 컴파일된 진짜 Postgres라 트리거도 int8도 그대로다.

## 배포는 npm이다. 번들이 아니다

CLI는 **npm에 배포되고**, 두 플러그인(claude-code, codex) 모두 스킬만 담는다.
에이전트는 `npx @holiday-cfo/cli@latest`로 CLI를 얻는다. 플러그인 안에 커밋된
바이너리는 없다 — [ADR-007](/docs/dev/adr).

처음엔 반대였다. 플러그인에 단일파일 번들(`plugins/claude-code/bin/holiday.mjs`)을 커밋했고, 그게
`better-sqlite3`가 아니라 **`node:sqlite`**를 고른 이유였다: 네이티브 애드온은 번들이
안 되니 설치 스텝 없는 플러그인에 넣을 수 없었다. npx로 옮기면서 그 제약은 사라졌지만
결론은 그대로다 — `node:sqlite`는 여전히 의존성이 0이고 네이티브 빌드가 없어서 `npx`
첫 실행이 가볍다. 근거가 "번들 불가"에서 "설치가 가벼움"으로 바뀌었을 뿐이다.

`store-sqlite`가 `store-postgres`와 스토어를 공유하는 것도 여기서 값을 한다: 대시보드
프로젝트는 어차피 `npm install`을 하므로, 라이브 원장이 필요해지면 같은 스토어에
Supabase 어댑터를 물리면 된다.

## 감사

`audit_log`가 해시 체인이다. 각 행이 `prev_hash`로 직전 행에 커밋하고, `txn_append`
행은 **거래 id가 아니라 내용의 sha256에 커밋한다.** 손으로 전기를 고치면 잔액은 맞고
통화도 맞고 identity weight도 맞아서 다른 모든 검사를 통과하는데, **체인만 알아챈다.**

**정직한 한계:** 우발적 변조는 잡지만 쓰기 권한을 가진 자가 체인 전체를 재계산하는 건
못 막는다. head를 파일 **밖에** 앵커해야 닫힌다 — `holiday verify --head`가 그 용도다.
