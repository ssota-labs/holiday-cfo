## 패키지 구조

```
                  @holiday/core
              (도메인 + 포트 + 유스케이스)
                        ▲
        ┌───────────────┴───────────────┐
        │                               │
  @holiday/store-sql              store-testkit
  (LedgerStore 구현 — 하나)        (적합성 스위트)
        ▲                               ▲
   ┌────┴─────┐                         │
   │          │                         │
store-sqlite  store-postgres            │
(드라이버·     (드라이버·                │
 스키마·        스키마·                  │
 PRAGMA)        plpgsql)                 │
   ▲          ▲                         │
   └────┬─────┴─────────────────────────┘
        │
   @holiday/cli
   (composition root)
```

| 패키지 | 역할 |
|---|---|
| `@holiday/core` | 도메인·포트. 어댑터 import 금지 |
| `@holiday/store-sql` | `LedgerStore` 구현 단일본. Dialect로 방언 접음 |
| `@holiday/store-sqlite` | SQLite 드라이버·스키마·마이그레이션 |
| `@holiday/store-postgres` | Postgres 드라이버·스키마·마이그레이션 |
| `@holiday/store-testkit` | `runLedgerStoreConformance()` |
| `@holiday/cli` | composition root. 엔진 선택 |

`@holiday/ports` 패키지는 두지 않는다. 포트 인터페이스는 도메인 소유다.

## 런타임

| 경로 | 엔진 | 비고 |
|---|---|---|
| 제품 CLI (`holiday init` → `openLedger`) | **SQLite만** | `workspace.ts`의 `store: 'sqlite'`. `sqliteLedgerStore` 고정 |
| 라이브러리 / 테스트 | Postgres 가능 | `store-postgres` + in-process **pglite** (Postgres-in-WASM). 목(mock)이 아님 |

CLI 바이너리 경로에 Postgres/Supabase 연결 옵션은 없다. Postgres 엔진 패키지는
존재하며 적합성 스위트로 검증한다. 제품 SoR 경로는 로컬 SQLite다.

정의가 갈라진다.

## assertEngineTier

`init()`에서 `tier === 'engine'`이면 아래 세 필드가 모두 true여야 한다. 아니면
`TierContractError`를 throw한다.

<SchemaTable
  name="StoreCapabilities"
  kind="port"
  source="packages/core/src/ports/ledger-store.ts"
  fields={[
    { name: 'tier', type: "'engine' | 'projection'", required: true, description: 'engine만 system of record' },
    { name: 'atomicMultiRowWrite', type: 'boolean', required: true, invariant: 'engine이면 true', description: '다행 all-or-nothing' },
    { name: 'uniqueConstraints', type: 'boolean', required: true, invariant: 'engine이면 true' },
    { name: 'readAfterWriteConsistency', type: 'boolean', required: true, invariant: 'engine이면 true' },
    { name: 'serverSideAggregation', type: 'boolean', description: 'false면 streamPostings fold' },
    { name: 'enforcesInvariantsAtRest', type: 'boolean', description: '트리거/CHECK' },
  ]}
/>

원자 multi-row write·유니크·read-after-write가 없는 백엔드는 engine을 선언할 수
없다. 설계상 projection이다. 논거는 [ADR](/docs/dev/adr).

`streamPostings()`는 필수 프리미티브다. `getBalances()`는 capability 게이트 빠른
경로다.

## 불변식 4중 링

| 링 | 위치 | 역할 |
|---|---|---|
| 1. 도메인 | `Txn.create()` | 순수·동기. 불균형 `ValidatedTxn` 생성 불가 |
| 2. 포트 | `appendTxn()` | 원자성·내구성 |
| 3. 저장소 | 트리거 11개 | SQLite `RAISE(ABORT)` / Postgres plpgsql. engine 공통 |
| 4. 감사 | `holiday verify` | 전체 스캔 + 해시 체인 |

포트는 의도적으로 가장 약한 링이다. 명세 대상은 기계적 보장이지 비즈니스 규칙이
아니다.

트리거 11개 (마이그레이션 `…_invariant_triggers`):

1. `txn_seal_requires_balance`
2. `posting_rejects_placeholder_account`
3. `posting_commodity_conformance`
4. `posting_identity_weight`
5. `posting_immutable_insert`
6. `posting_immutable_update`
7. `posting_immutable_delete`
8. `txn_never_unseals`
9. `commodity_exponent_immutable`
10. `audit_log_immutable_update`
11. `audit_log_immutable_delete`

## 적합성 스위트

계약의 실행 가능 형태는 `runLedgerStoreConformance()`다. **케이스 18개**, 엔진마다
실행한다.

대표 케이스:

- unit of work 중간 예외 → 0행
- 캐스트로 밀어넣은 불균형 → 저장소 거부
- `2^53+1` 왕복이 `number`로 손실되지 않음
- draft 제외 / promote 후 포함
- idempotency 키 재생·충돌

## 플러그인 배포

플러그인은 설치 시 빌드하지 않는다. 커밋된 산출물이 실행 파일이다.

- 런타임: Node builtin **`node:sqlite`** (네이티브 애드온 없음)
- 번들: `plugin/bin/holiday.mjs` (커밋본). CI가 재번들 후 diff 검사

## 감사

`audit_log`는 해시 체인이다. `txn_append`는 거래 id가 아니라 **내용 sha256**에
커밋한다. 잔액이 맞아도 내용이 바뀌면 `content_tampered`다.

한계: 쓰기 권한자가 체인 전체를 재계산하는 공격은 막지 못한다. head를 파일 밖에
앵커한다 (`holiday verify --head`).
