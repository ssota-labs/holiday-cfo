## 한 줄

사용자는 장부 화면의 **페이지와 배치**를 마크다운으로 마음대로 짜고, 그 안의
금액·잔액·현금 여유는 항상 장부에서 뽑아 둔 **스냅샷**만 본다.

## 문제

지금 장부 화면은 한 장의 배치 파일로만 블록을 고른다. 블록 목록은 닫혀 있어
잘못된 금액을 넣을 수는 없지만, 다음이 답답하다.

- 여러 페이지(개요 / 카드 / 세금 메모)로 나누기 어렵다
- 문장과 표를 섞어 “내 상황 설명 + 숫자 블록”을 한 흐름으로 쓰기 어렵다
- 배치 문법이 JSON이라, 사람이 읽고 고치기보다 에이전트 전용에 가깝다

문서 사이트는 이미 마크다운 + 커스텀 칸으로 잘 돌아간다. 장부 화면도 그
엔진으로 통째로 바꾸자는 제안이 나오기 쉽다. 다만 문서 사이트는
**정책·기획을 빌드하는 도구**이고, 장부 화면은 **내 돈 스냅샷을 보여주는
표면**이다. 엔진을 그대로 옮기면 무게·배포·숫자 안전이 한꺼번에 흔들린다.

필요한 것은 문서 사이트의 **작성 느낌**(페이지를 파일로 두고, 허용된
칸만 끼워 넣기)이다. 문서 사이트 자체를 장부 화면으로 쓰는 일이 아니다.

## 무엇을 저장·보여 주는가

장부 DB 스키마는 바꾸지 않는다. 사용자 프로젝트의 화면 폴더에 남는 것이다.

| 사람이 읽는 이름 | 예시 | 왜 필요한가 |
|---|---|---|
| 화면 페이지 | `pages/index.mdx`, `pages/cards.mdx` | 내가 보고 싶은 순서·구성 |
| 숫자 스냅샷 | 잔액·현금 여유·장부 건강이 든 데이터 파일 | 숫자는 장부에서만 온다 |
| 허용 칸 | 현금 여유표, 잔액표, 분류 대기 목록 등 | 금액 칸을 사람이 직접 못 넣게 막는다 |
| 짧은 메모 | “이번 달 카드 한도 줄임” 같은 문장 | 숫자 없이 맥락만 |

## 목표

1. 사용자는 화면을 **여러 페이지**로 나누고, 페이지마다 문장과 허용 칸을
   섞어 쓴다.
2. 금액·잔액·런웨이·부족 구간은 **스냅샷에서만** 읽는다. 페이지 본문에
   금액을 직접 적어 넣는 경로는 없다(적어 넣어도 반영·갱신되지 않거나
   검증에서 막는다).
3. 로컬에서는 분류 대기를 클릭으로 처리한다. 배포본은 읽기 전용
   사진이다.
4. 문서 사이트 엔진(전체 문서 UI·사이드바·정적 문서 빌드 파이프라인)을
   장부 화면 런타임으로 **이식하지 않는다.** 작성 모델만 맞춘다.
5. 에이전트/사용자가 만지는 어휘는 닫힌 칸 목록이다. 새 칸 이름은 렌더되지
   않고 실패한다.

## Non-goals

- 라이브로 장부 DB를 브라우저가 직접 여는 웹 앱
- 문서 사이트와 장부 화면을 한 앱·한 라우트로 합치기
- 금액 prop·쓰기 액션을 페이지 작성 어휘에 노출하기
- 카드사·세액·할부 공식을 화면이 다시 계산하기
- “웹 대시보드를 제품의 본체”로 만들기 (본체는 여전히 CLI + 채팅 shell-out)

## 비기능

- 스냅샷 금액은 문자열(소수 표기)로 끝까지 간다. 화면이 `number`로 왕복하지
  않는다.
- 스냅샷을 굽는 시점의 잔액·현금흐름·검증은 서로 어긋나면 안 된다(한 묶음으로
  읽는다).
- 현금 여유의 **부족 구간**은 숨기지 않는다.
- 화면 패키지 버전은 CLI와 같은 태그로 맞춘다.
- 기존 JSON 배치가 있으면 한동안 같이 읽거나, 분명한 이전 경로를 준다.

## 성공 지표

- 사용자는 두 페이지 이상인 화면을 파일만으로 구성한다
- 페이지에 금액을 직접 적어도, 화면에 권위 있는 숫자로 나타나지 않는다
- 스냅샷을 다시 구우면 페이지 구조는 그대로이고 숫자만 바뀐다
- 로컬에서 분류 대기 한 건을 클릭으로 처리한다
- 배포된 화면에는 쓰기 UI가 동작하지 않는다
- 문서 사이트 의존 스택이 사용자 화면 폴더의 런타임으로 끌려가지 않는다

## 결과 계약

장부 화면은 **스냅샷 뷰어**다. 페이지 파일은 배치와 설명만 맡고, 숫자는
스냅샷만 맡는다. 기계 계약은
[SPEC-dash-mdx-shell](/docs/spec/dash-mdx-shell)과
[ADR-012](/docs/development/adr/adr-012)에 둔다.
