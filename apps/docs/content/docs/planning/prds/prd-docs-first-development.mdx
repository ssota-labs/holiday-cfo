## 문제

현재 저장소는 제품 문서가 단일 진실이라고 선언하지만, 구현 전에 PRD·스펙·구현계획을
확정하도록 막는 절차가 없다. 그 결과 코드가 먼저 바뀌고 문서가 나중에 역기록될 수 있다.

## 목표

- 추가 개발의 이유와 범위를 PRD에서 먼저 합의한다.
- PRD·스토리·기능 수락 기준은 **기획자·비개발 팀원이 읽고 판단**할 수 있게 쓴다
  (저장·조회 데이터를 사람 말로, 개발 용어만으로 본문을 때우지 않는다 —
  [workflow](/docs/planning/workflow)).
- 기획 한글 본문은 상태 올리기 전에 **항상** im-not-ai(`/humanize`)로 윤문한다.
- 사용자와 시스템이 관찰하는 계약을 스펙으로 확정한다.
- 영향 범위·구현 순서·검증 방법을 구현계획으로 남긴다.
- 구현 PR은 기본 브랜치에 이미 존재하는 준비된 구현계획만 참조한다.
- 에이전트 지침과 CI가 같은 절차를 강제한다.

## Non-goals

- 문서의 제품 판단을 자동으로 대신하지 않는다.
- 모든 유지보수에 새 PRD와 스펙을 만들지 않는다.
- 긴급 변경을 위한 무제한 우회 절차를 만들지 않는다.
- 기존 완료 PRD와 CLI 스펙을 새 형식으로 일괄 재작성하지 않는다.

## 변경 유형

| 유형 | 필요한 문서 |
|---|---|
| `product` | PRD + 스펙 + 구현계획 |
| `bugfix` | 기존 PRD·스펙 링크 + 구현계획 |
| `maintenance` | 구현계획 |
| docs-only | 기획 게이트 면제 |

## 성공 지표

- 코드 변경 PR에 base 브랜치의 `ready` 구현계획이 없으면 CI가 실패한다.
- 같은 PR에서 구현계획과 코드를 함께 추가해 선행 절차를 우회할 수 없다.
- PRD·스토리·스펙·구현계획의 깨진 참조와 중복 ID를 자동 검출한다.
- 저장소 기본 테스트가 기획 문서 검증을 포함한다.
