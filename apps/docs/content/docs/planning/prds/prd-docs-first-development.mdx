## 한 줄로

제품·버그·유지보수 변경은 **기획 문서가 기본 브랜치에 먼저** 올라간 뒤에만
구현 PR을 연다. 코드와 문서 순서가 뒤집히지 않게 CI와 에이전트 지침이
같은 규칙을 강제한다.

## 문제

제품 문서를 기준이라 해도, 구현 전에 PRD·스펙·구현계획을 확정하도록
**막는 절차가 없었다**. 코드가 먼저 바뀌고 문서가 나중에 따라오면
다음이 흔들린다.

- 왜 이 기능을 만드는지
- 사용자에게 무엇이 남고 보이는지
- 어떻게 검증하는지

문서를 “나중에 채우면 된다”로 두면 리뷰는 디프만 보고, 계약은 남지 않는다.

## 무엇을 남기고 막는가

### 1) 남기는 문서

| 사람이 읽는 이름 | 무엇을 적는가 | 왜 필요한가 |
|---|---|---|
| PRD | 문제·목표·비목표·성공 지표 | 범위와 이유를 먼저 합의 |
| User Story | 누가·무엇을·왜, 수락 기준 | 관찰 가능한 결과로 체크 |
| Spec | CLI·화면·기계 계약 | 구현이 맞출 경계 |
| Implementation Plan | 영향 범위·순서·검증 | 어디에 손대고 어떻게 확인할지 |

작성·독자 기준은 [기획 프로세스](/docs/workflow/planning)에 둔다.
기획 한글 본문은 상태를 올리기 전에 im-not-ai(`/humanize`)로 윤문한다.

### 2) 막는 우회

| 막히는 일 | 결과 |
|---|---|
| 코드 PR에 기본 브랜치의 준비된(`ready`) 구현계획이 없음 | CI 실패 |
| 같은 PR에서 구현계획과 코드를 한꺼번에 추가 | CI 실패 (선행 우회 불가) |
| 기획 문서 ID 중복·깨진 링크 | 저장소 테스트 실패 |

docs-only 변경은 이 게이트에서 뺀다. 그 밖의 **무제한 긴급 우회**는 두지 않는다.

## 목표

- 추가 개발의 이유와 범위를 PRD에서 먼저 합의한다
- 사용자·시스템이 관찰하는 계약을 스펙으로 확정한다
- 영향 범위·구현 순서·검증 방법을 구현계획으로 남긴다
- 구현 PR은 기본 브랜치에 이미 있는 준비된 구현계획만 참조한다
- 에이전트 지침(`AGENTS.md`)과 CI가 같은 절차를 강제한다

## Non-goals

- 문서의 제품 판단을 자동으로 대신하지 않는다
- 유지보수마다 새 PRD와 스펙을 요구하지 않는다
- 긴급 변경용 무제한 우회 절차는 두지 않는다
- 끝난 PRD·CLI 스펙을 새 형식으로 일괄 다시 쓰지 않는다

## 변경 유형

| 유형 | 필요한 문서 |
|---|---|
| `product` | PRD + 스펙 + 구현계획 |
| `bugfix` | 기존 PRD·스펙 링크 + 구현계획 |
| `maintenance` | 구현계획 |
| docs-only | 기획 게이트 면제 |

## 포함 스토리

- [us-docs-first-development](/docs/planning/stories/us-docs-first-development)

## 비기능

- 검증은 참조 그래프·PR base SHA·문서 ID에 한정한다 (제품 판단은 사람 리뷰)
- 원장 데이터·공개 CLI 계약은 이 절차만으로 바꾸지 않는다

## 성공 지표

1. 코드 변경 PR에 base 브랜치의 `ready` 구현계획이 없으면 CI가 실패한다
2. 같은 PR에서 구현계획과 코드를 함께 추가해 선행을 우회할 수 없다
3. PRD·스토리·스펙·구현계획의 깨진 참조와 중복 ID를 자동으로 잡는다
4. 저장소 기본 테스트에 기획 문서 검증이 포함된다
