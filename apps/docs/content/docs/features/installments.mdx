## 개요

할부는 구매일에 부채 전액을 세우고, 현금만 N개월에 걸쳐 나가는 스케줄이다. 잔액은 카드 일반 계정이 아니라 `Liabilities:Card:<카드>:Installment`(또는 `--liability`)에 둔다.

일반 청구는 "사용기간 내 posting 합"이다. 할부 구매 posting이 카드 일반 계정에 있으면 첫 청구서가 전액으로 부푼다. [POLICY-003](/docs/domain/policy#policy-003)

## 동작

`installment add` 시:

1. 카드에 청구주기가 있어야 한다. 없으면 회차 날짜를 계산할 수 없다.
2. 부채 계정이 없으면 `<card>:Installment`를 자동 생성한다. `cash: false`, 부모는 카드 계정.
3. 구매일에 `Expenses … Dr` / `Installment … Cr`로 전액 기표한다.
4. `buildInstallmentSchedule`이 카드 결제일 기준으로 회차를 만든다. 1회차는 구매가 속한 청구서의 결제일.
5. `--fees`가 없으면 무이자(`feeMinor = 0`). 있으면 회차 수와 같은 개수의 관측 수수료를 붙인다.

원금 행의 합은 구매금액과 정확히 같다. 나머지는 기본으로 1회차에 붙인다. [POLICY-007](/docs/domain/policy#policy-007)

할부수수료는 계산하지 않는다. 명세서에서 읽은 값만 `--fees`로 받는다. [POLICY-006](/docs/domain/policy#policy-006) 수수료는 원금 합 불변식 밖이다. [POLICY-009](/docs/domain/policy#policy-009)

`installment revise`는 명세서가 말하는 회차·날짜·원금·수수료로 스케줄을 덮어쓴다. 입력 플래그는 `--data`다. 전역 `--json`(기계 출력)과 이름이 겹치지 않도록 분리했다. 원금 합이 구매금액과 다르면 거부한다.

`cashflow`는 스케줄 행만 투영한다. 구매 posting은 이미 구매일 잔액에 반영되어 있다.

## 제약

- 카드에 `card add`로 청구주기가 있어야 한다.
- `--fees` 개수 = `--months`. 음수 수수료 거부.
- `revise`의 원금 합은 구매 `totalMinor`와 일치해야 한다. 수수료를 원금에 넣지 않는다.
- 청구주기가 사라진 카드의 할부는 `cashflow`에서 orphan 경고와 함께 제외된다.
