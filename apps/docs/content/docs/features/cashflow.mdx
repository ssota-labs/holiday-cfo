카드·할부·정기지출·정기수입·대출을 날짜별로 쌓아, **어느 날 통장이 버티는지**를
본다. 장부에 이미 찍힌 잔액에서 시작해, 아직 안 온 일정만 앞으로 투영한다.

## 수락 기준

1. 시작 잔액은 `--cash` 계정 합이다
2. 카드·할부·정기·대출 투영이 정책을 지킨다
   ([POLICY-003](/docs/domain/policy#POLICY-003) · [004](/docs/domain/policy#POLICY-004) ·
   [005](/docs/domain/policy#POLICY-005) · [008](/docs/domain/policy#POLICY-008) ·
   [010](/docs/domain/policy#POLICY-010))
3. 날짜별 잔액이 음수면 **부족**으로 표시한다
4. 조회만 한다 — 장부 숫자를 바꾸지 않는다
5. `--spend` / `--receive`는 가정일 뿐이다. 전표를 만들지 않는다

## 다섯 스케줄 → 런웨이

| | 모양 | 계정 |
|---|---|---|
| **카드 청구주기** | 회전 — 사용기간이 닫히고 결제일에 전액 | `Liabilities:Card:*` |
| **할부** | 고정 N개월 분할 | `Liabilities:Card:*:Installment` (별도) |
| **대출** | 분할상환 (원금+이자) | `Liabilities:Loans:*` |
| **정기지출** | 반복 유출 | 계정 아님 (스케줄 테이블) |
| **정기수입** | 반복 유입 | 계정 아님 (스케줄 테이블) |

**스케줄에서 원장으로 가는 화살표가 없다.** 예측은 전기되지 않는다 — 금리가
바뀌거나 구독을 해지할 때마다 과거가 오염된다.

현금이 뭔지는 `account.cash` 플래그가 정한다 — prefix가 아니다 ([ADR-004](/docs/development/adr/adr-004)).

## 실제 출력

```
현재 현금 (2026-07-17):  350000000 KRW

2026-07-25   +    2200000   →     352200000
             급여                                  +3000000
             월세                                   800000
2026-08-01   -      400000   →     351800000
             신한  2026-06-15..2026-07-14           400000
2026-08-25   +     732948   →     352532948
             급여                                  +3000000
             월세                                   800000
             KB 주담대 (1/360)                      1467052
2026-09-01   -      100000   →     352432948
             냉장고 (1/12)                           100000
```

8/25에 급여·월세·주담대가 **같은 날 합류**한다. 실제 통장이 하는 일 그대로다.

`⚠ 부족`이 답이다. 잔액은 숫자 하나를 주지만 현금흐름은 **어느 날 터지는지**를 준다.

## 가정은 장부를 건드리지 않는다

"이 대출 받으면?", "집 사면?" 같은 가정은 **런웨이에 얹어서** 계산한다. 투기적 거래를
넣었다 지우는 게 아니다 — `cashflow`가 가정을 직접 받는다.

```bash
holiday cashflow --until 2027-06-30 \
  --spend "2026-09-01 5000000 새 노트북" \
  --receive "2026-12-25 3000000 보너스"
```

장부는 그대로다. `--spend` 없이 다시 돌리면 가정이 사라진다.
