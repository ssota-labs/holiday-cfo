## 개요

대출 모듈은 부채 계정에 상환 스케줄을 붙인다. 스케줄은 예측이며 원장에 전기되지 않는다. 실제 상환은 `loan pay`가 스케줄 회차로 원금·이자를 나눠 3-leg 거래를 기표한다.

명세서에 "총액 N원 납부"만 있을 때, 원금/이자 분해는 스케줄이 담당한다.

방법:

| `--method` | 의미 |
|---|---|
| `annuity` (기본) | 원리금균등 |
| `equal_principal` | 원금균등 |
| `bullet` | 만기일시상환(이자 매월, 원금 만기) |
| `interest_only` | 거치(이자만). 원금 상환 없음 |

## 동작

`buildLoanSchedule`이 회차별 개시잔액·원금·이자·종료잔액을 만든다. 이자 반올림은 0에서 멀어지는 쪽이다. [POLICY-015](/docs/domain/policy#policy-015)

원금 행의 합은 대출 원금과 같다. 마지막 회차가 잔여를 흡수한다. [POLICY-012](/docs/domain/policy#policy-012) `interest_only`만 예외로 원금 합 0이다.

`loan check`는 as-of 기준 스케줄 종료잔액과 원장 부채 잔액(부호 보정)을 비교한다. 차이는 미납·중도상환·미기록 금리변경·수수료 등의 신호다. [POLICY-011](/docs/domain/policy#policy-011)

`loan pay`는 `--date`에 해당하는 회차가 있어야 한다. `--amount`가 스케줄과 다르면 이자는 스케줄값을 유지하고 차액을 원금에 반영한다. 실납이 스케줄 이자보다 작으면 거부한다.

`cashflow`의 `projectLoans`는 `dueDate > today`인 회차만 투영한다.

### 두 가지 거부

1. **0% annuity** — 원리금균등 공식의 분모 `((1+r)^n − 1)`이 0이 된다. 0%면 `equal_principal`을 쓰라고 거부한다. [POLICY-014](/docs/domain/policy#policy-014)

2. **interest_only balloon 없음** — 거치 스케줄은 원금을 0으로만 둔다. 만기 일시상환(balloon)을 지어내지 않는다. 출구는 별도 계약·별도 스케줄이다. [POLICY-013](/docs/domain/policy#policy-013)

## 제약

- 스케줄은 예측이다. 금리·중도상환 변경 시 원장을 자동 재작성하지 않는다. `loan check`로 괴리를 본다.
- `interest_only`는 balloon을 만들지 않는다. [POLICY-013](/docs/domain/policy#policy-013)
- 0% `annuity`는 거부한다. [POLICY-014](/docs/domain/policy#policy-014)
- `loan pay`의 `--date`는 스케줄 `dueDate`와 정확히 일치해야 한다.
