CLI는 **OCR을 하지 않는다.** 에이전트의 vision 모델이 이미지를 읽고 파싱된 JSON을
넘기면, 여기서 새 것인지 판단하고, 리뷰 게이트에서 맞는지 판단한다.

그 분리가 중요하다. **모델은 이 시스템에서 가장 신뢰도가 낮은 부품**이고, 여기 있는
전부가 모델의 실수를 되돌릴 수 있게 유지하려고 존재한다.

## 드래프트지 별도 staging이 아니다

수집은 `status='draft'`인 **진짜 거래**를 쓴다. 그래서 균형 규칙이 수집 시점에
돌고, **불균형 드래프트는 만들 수조차 없다.**

두 가지가 따라온다. 망가진 항목을 큐에 넣어두고 남에게 떠넘길 수 없고, 나중의 승인이
**실패할 수 없는 status flip**이 된다.

**`--idem-key`는 선택이 아니다.** CLI를 shell out하는 에이전트는 타임아웃 때 **반드시**
재시도한다. 같은 키 + 같은 요청은 replay하고, 같은 키 + 다른 요청은 하드 에러다.
이중 기표는 원장 신뢰를 파괴하는 유일한 실패 모드다.

## 중복 판단은 권위가 층으로 나뉜다

원래 계획은 자연키(`계정‖날짜‖금액‖상호`)에 `UNIQUE`를 걸어 차단하라면서, 동시에
"같은 날 같은 카페 ₩4,500 두 잔은 평범한 화요일"이라고 했다. **둘 다 참일 수 없다** —
그 두 잔은 자연키가 정확히 같고, 차단하면 진짜 거래를 거부한다.

| 근거 | 판단 | 왜 |
|---|---|---|
| **이미지 sha256** | 차단 | 같은 바이트는 항상 같은 캡쳐다. 틀릴 수가 없는 유일한 검사 |
| **externalRef** | 차단 | 카드사 거래ID보다 카드사 기록에 대해 권위 있는 건 없다 |
| **자연키** | **경고** | 추측이다. 그리고 커피를 두 번 사는 바로 그때 틀린다 |
| **±3일 퍼지** | **경고** | 카드 앱과 명세서가 날짜를 다르게 말한다 |

상호명 정규화는 **의도적으로 보수적**이다. 카드사가 붙이는 법인 형태(`(주)`, `주식회사`)와
네트워크 구분자는 떼지만, **지점은 안 뗀다** — `GS25 역삼점`과 `GS25 강남점`은 다른
곳이고, 합치면 진짜 구매 둘이 하나로 보인다.

## 거부해도 기록은 남는다

거부된 항목은 **삭제하지 않는다.** 그건 중복 판단의 기억이고, 지우면 같은 캡쳐가 영원히
다시 제안된다.

```bash
holiday review list
holiday review accept <id>              # 균형은 이미 잡혀 있으니 실패할 수 없다
holiday review reject <id> --reason "…"  # 기록은 남는다
```

## 게이트는 추측 허가증이 아니다

리뷰가 잡는 건 **잘못된 카테고리**지 **오독한 금액**이 아니다. `₩1,240,00`을 사람이
승인하면 틀린 채로 승인된다.

그래서 스킬이 "금액이 불분명하면 멈추고 물어라"에 지면을 쓴다 — 게이트가 생겼다고
모델이 느슨해지면 게이트가 없느니만 못하다.

## BYOC 팩스 경로 (벤더 비종속)

은행/카드 **팩스 자동통지**는 SOLAPI FAX-RECEIVE 같은 가상팩스 → 사용자가 고른
HTTPS 엔드포인트로 들어온다. holiday는 관리형 멀티테넌트 웹을 운영하지 않는다
(`holiday deploy`가 스캐폴드만 한다).

```
KB 팩스 → SOLAPI → (Vercel+Supabase | ChatGPT Sites) → Cursor Automation → holiday MCP → draft → review
```

- **계약:** `@holiday-cfo/contracts` (금액은 decimal string). CLI · HTTP · MCP가 같은
  usecase(`submitIngest` / `review*` / `verifyLedger`)를 호출한다.
- **Vercel+Supabase:** Postgres는 engine tier (`prepare: false` + bigint). `fileId`
  UNIQUE, 5초 내 ack, `after()` + `/api/fax/recover`.
- **ChatGPT Sites:** vinext + D1/R2. Queue/Cron 없음 → 실패는 D1에 남기고 UI/MCP로
  재처리. D1은 interactive `unitOfWork`가 없어 **engine 원장으로 선언하지 않는다**
  (inbox + review/export만). `holiday deploy check` / `@holiday-cfo/store-d1` 게이트.
- **보존:** 원본은 private Storage/R2. `.holiday/**` · `*.db` · `.env*`는 배포 산출물에
  넣지 않는다.
- **비용:** Cursor Automation은 파싱 실패 시에만. 배포 스킬은 preview에서 합성 팩스
  중복 테스트를 통과한 뒤에만 production을 제안한다.
