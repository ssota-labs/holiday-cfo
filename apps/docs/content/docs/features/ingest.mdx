## 개요

`holiday ingest`는 이미 파싱된 거래 JSON을 원장에 **드래프트**로 넣는다. CLI는 OCR을 수행하지 않는다. 이미지 해석은 에이전트 vision 모델(또는 사람)의 몫이다.

흐름:

1. 파서가 `--data`로 submission JSON을 넘긴다.
2. `ingest submit`이 균형 검사를 통과한 거래를 `status='draft'`로 기록한다.
3. `review list` → `accept` / `reject`로 사람이 결정한다.
4. 승인된 항목만 잔액·현금흐름·마감에 포함된다.

드래프트는 별도 staging 테이블이 아니라 **진짜 거래**다. 불균형 드래프트는 생성되지 않는다. 승인은 실패할 수 없는 status flip이다.

## 동작

### 입력 계약

`--data`에 JSON을 넣는다. 전역 `--json`(기계 출력)과 이름이 겹치지 않도록 `--data`를 쓴다. `--json`으로 submission을 넘기지 않는다.

항목은 flat 금액이 아니라 **legs**다. 카테고리 판단도 같은 JSON에 포함된다. 스키마는 `packages/cli/src/ingest.ts`의 `INGEST_SUBMISSION`이다.

### 중복

| 근거 | 효과 |
|---|---|
| 이미지 sha256 (`--image` 또는 `sourceSha256`) | 동일 바이트면 차단 (`duplicate_image`) |
| `externalRef` | 동일 발행사 거래 ID면 차단 (`duplicate_external_ref`) |
| 자연키(계정·날짜·금액·상호) | 경고만. 동일 구매 두 번은 허용 |
| ±3일 near-duplicate | 경고만 |

### idem-key

`--idem-key`는 코드상 선택이다. 같은 키 + 같은 request SHA면 저장된 응답을 replay한다. 같은 키 + 다른 요청은 거부한다. 에이전트가 타임아웃 후 재시도할 때 이중 기표를 막으려면 키를 넣는 것이 권장된다.

### 리뷰

- `accept`: draft → posted. ingest item → accepted.
- `reject --reason`: draft 거부. 항목 행은 **삭제하지 않는다**(중복 기억).

마감(`close`)은 기간 내 draft가 남아 있으면 막힌다.

## 제약

- CLI에 OCR 경로가 없다. 이미지는 `--image`로 해시만 한다.
- 드래프트는 잔액·리포트에서 제외된다. 마감 게이트는 미해결 draft를 거부한다.
- weight 합은 정확히 0이어야 한다. [POLICY-001](/docs/domain/policy#policy-001)
- submission은 `--data`로만 전달한다. `--json`은 출력 모드다.
