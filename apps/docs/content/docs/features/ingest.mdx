은행·카드 내역을 **이미 파싱한 JSON**으로 넘긴다. CLI는 OCR을 하지 않는다.

기본은 **분류 대기(draft)** 전표다. 넣는 순간에도 균형 규칙은 돌아가서, 안 맞는
항목은 만들 수조차 없다. `--post`는 검토 큐를 건너뛰고 바로 확정한다 — 과거
내역을 한꺼번에 넣을 때용이다. 일상 흐름은 사람이 승인·반려한다.

## 수락 기준

1. 기본 submit은 분류 대기로 남는다 (`--post`면 바로 확정)
2. 같은 이미지/소스 해시·external_ref는 다시 넣지 못한다
3. 반려한 행도 남겨 두어 중복을 기억한다
4. 승인해 확정된 전표만 잔액에 들어간다
5. 마감은 그 기간의 분류 대기가 있으면 막는다
6. `--data`와 `--data-file` 중 하나만 쓴다
7. `--idem-key`로 같은 요청을 안전하게 재시도한다

## 중복 판단

| 근거 | 판단 |
|---|---|
| 이미지/소스 sha256 | 차단 |
| externalRef | 차단 |
| 자연키 | 경고만 |
| ±3일 퍼지 | 경고만 |

반려한 항목은 삭제하지 않는다 — 중복 기억이다.

## BYOC 팩스 경로 (벤더 비종속)

은행/카드 **팩스 자동통지**는 SOLAPI FAX-RECEIVE 같은 가상팩스 → 사용자가 고른
HTTPS 엔드포인트로 들어온다. 관리형 멀티테넌트 웹은 없다
(`holiday deploy`가 스캐폴드만 한다).

```
KB 팩스 → SOLAPI → (Vercel+Supabase | ChatGPT Sites) → Cursor Automation → holiday MCP → draft → review
```

- **계약:** `@holiday-cfo/contracts` (금액은 decimal string). CLI · HTTP · MCP가 같은
  usecase(`submitIngest` / `review*` / `verifyLedger`)를 호출한다.
- **Vercel+Supabase:** Postgres는 engine tier (`prepare: false` + bigint). `fileId`
  UNIQUE, 5초 내 ack, `after()` + `/api/fax/recover`.
- **ChatGPT Sites:** vinext + D1/R2. Queue/Cron 없음 → 실패는 D1에 남기고 UI/MCP로
  재처리. D1은 interactive `unitOfWork`가 없어 **engine 원장으로 선언하지 않는다**
  (inbox + review/export만). `holiday deploy check` / `@holiday-cfo/store-d1` 게이트.
- **보존:** 원본은 private Storage/R2. `.holiday/**` · `*.db` · `.env*`는 배포 산출물에
  넣지 않는다.
- **비용:** Cursor Automation은 파싱 실패 시에만. 배포 스킬은 preview에서 합성 팩스
  중복 테스트를 통과한 뒤에만 production을 제안한다.
